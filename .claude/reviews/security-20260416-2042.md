# Security Review Report

## Summary

The app handles credentials reasonably well for a personal project: `EncryptedSharedPreferences` with AES256-GCM is used for storage, cleartext HTTP is explicitly disabled in the manifest, and no secrets are hardcoded. However, two issues need attention: the OkHttp logging interceptor unconditionally logs **full request/response bodies in all build variants** (including release), which will dump the Authorization header and full message content to Logcat; and the API key text field has no visual masking, displaying the key in plain text to anyone looking at the screen.

- **Location**: `app/src/main/java/com/openrouter/chat/ui/screens/settings/SettingsScreen.kt:212â€“217`
- **Issue**: The `OutlinedTextField` for the API key has no `visualTransformation` and no `keyboardOptions` with `KeyboardType.Password`. The key is shown in full whenever the Settings screen is open.
- **Why it matters**: The key is visible to shoulder-surfers and to any screenshot taken while the screen is active (e.g. by the Android task switcher). OpenRouter keys grant billable API access.
- **Suggestion**: Add `visualTransformation = PasswordVisualTransformation()` and `keyboardOptions = KeyboardOptions(keyboardType = KeyboardType.Password)` to the `OutlinedTextField`. Optionally add a toggle icon to reveal the key on demand.

**3. API key injected per-call in ChatViewModel instead of via a shared interceptor**

- **Location**: `app/src/main/java/com/openrouter/chat/ui/screens/chat/ChatScreen.kt:152â€“157`
- **Issue**: The chat streaming path builds a raw `OkHttpClient` request manually, hard-coding `https://openrouter.ai/api/v1/chat/completions` and attaching the auth header inline. The `getModels` Retrofit path has no auth header at all (see `OpenRouterApi.kt:12`).
- **Why it matters**: Auth logic is split across three places (AppModule interceptor, ChatViewModel inline, validateKey parameter). If the key source ever changes, all three must be updated. The `getModels` call will silently fail or use public-rate-limit tiers because it carries no auth. This is also a correctness bug.
- **Suggestion**: Add a single OkHttp interceptor in `AppModule` that reads the key from `SecurePreferences` and injects `Authorization: Bearer <key>` on every outgoing request. Remove the per-call header from `ChatViewModel` and the `@Header` parameter from `validateKey`.

### Minor suggestions đźźˇ

**4. `allowBackup="true"` in manifest**

- **Location**: `app/src/main/AndroidManifest.xml:9`
- **Issue**: Android's auto-backup will include the encrypted preferences file in cloud backups by default.
- **Why it matters**: The backup is encrypted by Android's backup key, so this is low risk â€” but if the device backup is compromised the `EncryptedSharedPreferences` file travels with it. The master key stored in the Android Keystore does **not** back up, so the backed-up file is not decryptable on another device. The real risk is that this gives a false sense that backups restore the API key (they won't).
- **Suggestion**: Add `android:fullBackupContent` or a `data-extraction-rules` XML that explicitly excludes `secure_prefs` from backup, to make the intent clear and prevent confusion.

**5. No minimum length check on the API key before the network call**

- **Location**: `app/src/main/java/com/openrouter/chat/ui/screens/settings/SettingsScreen.kt:71â€“72`
- **Issue**: `validateAndSaveApiKey` only checks `key.isEmpty()`. A single character passes through to the network validation call.
- **Why it matters**: Trivially avoidable network call; also, very short strings can indicate a paste error. OpenRouter keys follow a known prefix pattern (`sk-or-`).
- **Suggestion**: Add a minimum length guard (e.g. â‰Ą 20 characters) or a prefix check (`startsWith("sk-or-")`) before firing the validation request.
